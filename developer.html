INSIDE THE
MAGICAL MACHINE.

NpxMagicDoc의 정교한 설계와 톱니바퀴가 어떻게 맞물려 돌아가는지, 그 핵심 매커니즘을 탐구합니다.

0. 설계 철학

왜 '공장' 시스템인가요?

마크다운 변환은 변수가 너무 많습니다. 이미지가 없을 수도, 표가 너무 복잡할 수도, 혹은 Obsidian만의 독특한 문법이 섞여 있을 수도 있죠. 우리는 이를 해결하기 위해 **'6단계 자동화 공정'**을 도입했습니다. 각 부서(모듈)는 자신의 역할에만 충실합니다. 세척(전처리)하고, 품질 검사(검증)하고, 포장(엔진)하는 체계적인 흐름이 완벽한 결과물을 보장합니다.

1. 기술 스택

공장을 지탱하는 견고한 기술 기초입니다.

영역 사용 기술 선택 이유
언어 TypeScript 5.x 타입 안전성을 통해 대규모 문서 처리 중 발생할 수 있는 런타임 오류 방지
런타임 Node.js 18+ (ESM) 최신 비동기 I/O 매커니즘을 통한 빠른 이미지 및 파일 처리
문서 엔진 Pandoc / WeasyPrint 오픈 소스 생태계에서 가장 강력한 문서 변환 및 레이아웃 엔진
UI/UX Inquirer / Commander 사용자에게 복잡한 옵션을 직관적으로 물어보고 처리하는 CLI 표준

2. 아키텍처 흐름도

데이터 원석이 완제품이 되기까지의 전체 공정입니다.

STEP 01
CLI INTERFACE
사용자 주문 접수 및 인터랙티브 흐름 제어
STEP 02
MARKDOWN PREPROCESSOR
Obsidian 문법 정제 및 이미지 경로 자동 해결
STEP 03
CONTENT VALIDATOR
8가지 품질 기준 검사 및 자동 수정(Auto-fix)
STEP 04
STYLING SERVICES
동적 CSS 빌드, SVG 표지 생성, 폰트 서브세팅(최적화)
STEP 05
PANDOC ADAPTER
최종 가공 데이터를 Pandoc 명령어로 조합하여 렌더링
STEP 06
OUTPUT HANDLER
최종 파일 배송 및 임시 시설물 정리(Cleanup)

3. 개발 환경 구축

공방의 작업대를 세팅하고 마법을 개선하는 단계별 가이드입니다.

STEP 01

코드 복제 및 기초 공사

GitHub에서 설계도를 가져오고 필요한 부품(Library)들을 모읍니다.

git clone https://github.com/goodlookingprokim/markdown-to-document-cli.git
cd markdown-to-document-cli
npm install
STEP 02

실시간 빌드 시스템 가동

TypeScript를 Node.js가 이해할 수 있는 JavaScript로 실시간 변환합니다.

# 전체 빌드 수행
npm run build

# 수정 시 즉시 반영 (Watch 모드)
npm run dev
STEP 03

로컬 연결 및 시운전 (Link)

프로 팁: npm link를 사용하면 개발 중인 코드를 실제 설치된 패키지처럼 테스트할 수 있습니다. 

# 프로젝트 폴더에서
npm link

# 이제 어디서든 'm2d' 주문을 내려보세요!
m2d interactive

4. 배포 매커니즘

NPX는 어떻게 설치 없이 작동할까요?

NPX는 '임시 작업실'을 만드는 마법입니다. 사용자가 npx m2d를 입력하면, NPM은 온라인 저장소에서 우리 패키지를 찾아 **'임시 공간'**에 내려받습니다. 그리고 package.json의 bin 필드에 정의된 실행 파일을 즉시 호출하죠. 작업이 끝나면 임시 공간은 마법처럼 사라집니다. 

// package.json의 실행 파일 정의
"bin": {
  "m2d": "dist/cli.js",
  "markdown-to-document": "dist/cli.js"
}
